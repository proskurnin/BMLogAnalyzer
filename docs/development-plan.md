# План разработки

Этот документ фиксирует актуальное состояние после release `1.7.20` и текущего patch-цикла `1.7.21`: анализа НБС из архивов `13-07-2026.zip` и `kzp_logs_260715.zip`, доработок блока `Состав загрузки`, исправления регресса полноты обработки archive cache, добавления раздела устройств и привязки диагностик к `device_context`.

Ключевое правило сохраняется: факты из логов, гипотезы и рекомендации должны быть разделены. Нельзя делать вывод без evidence-строк с `source_file`, `line_number`, `timestamp` и `raw_line`.

## Реализовано

1. Точная корреляция NBS `Send Commands::info` и BM `Info, req/resp`:
   - извлекаются `rid`, `BmNumber`, `TmSerialNumber`, `tm serial number`, `reader id`;
   - BM Info сопоставляется только по совпавшему device id или при доказуемо единственном устройстве в окне;
   - при неоднозначности отчёт показывает статус `ambiguous` и не строит причинную цепочку;
   - в manifest добавлены интервалы `Send Commands::info -> BM Info req`, `BM Info req -> BM Info resp`, `BM Info resp -> QR`.

2. Сверка с выводами коллег по `kzp_logs_260715.zip`:
   - найдено 7 сценариев выхода НБС дольше 3 секунд;
   - проблемными классифицируются 5 сценариев;
   - среднее `MODE::VALIDATE -> первый QR` для проблемных сценариев: `81,702 сек`;
   - исключённые сценарии остаются в отчёте с причиной исключения.

3. Расширенная статистика `StopListDb` / `References_nbs_slm`:
   - считаются `count`, `min`, `max`, `average`, `median`, `p90`, `p95`;
   - выбросы выделяются относительно фактов того же сценария;
   - внешний норматив не используется;
   - max и выбросы выводятся с evidence-строками.

4. Сравнение по фактическим границам смены:
   - сценарии получают фазу `after_log_started`, `after_session_closed` или `unknown`;
   - HTML показывает сравнение по фазам только если есть фактические группы для сравнения.

5. Каталог QR-state маркеров:
   - фиксируются прямые маркеры `QR data`, пустой `QR data`, `qrData`, `settime`, `updateConfiguration`, очистка/сброс QR;
   - отчёт не пишет `QR пропал` без прямого marker evidence.

6. HTML и manifest:
   - в блок `Выход НБС в работу после открытия смены` добавлены классификация, device id, BM Info-корреляция, расширенная статистика StopListDb/References и QR-маркеры;
   - текстовый отчёт содержит факты, причины включения/исключения и evidence.

7. Блок `Состав загрузки`:
   - счётчик log-файлов в шапке синхронизирован с фактическим inventory архива;
   - размеры файлов показываются в `KB`, если значение меньше 1 MB;
   - список `Он содержит логи следующих типов` строится по фактическим log-файлам архива;
   - модальное окно по типу лога показывает список файлов этого типа, количество строк совпадает со счётчиком;
   - типы логов вынесены в единый справочник, используемый summary, manifest, section registry и HTML;
   - при полном покрытии сканером типы строятся по content/path-классификации файлов, при неполном покрытии используется archive inventory без двойного учёта;
   - в модалке по типу лога выводятся размер файла и причина классификации;
   - старые неполные archive cache игнорируются по версии cache manifest и архив распаковывается заново;
   - `Other log-like` файлы переопределяются по content-классификации, если сканер доказательно распознал ПО валидатора или операционную систему;
   - распакованные `.log.gz.log` сопоставляются с исходными `.log.gz` member-файлами архива, чтобы типы не терялись после gzip-распаковки;
   - `Evidence` в таблице распознавания типов переименован в `Признак распознавания`;
   - таблица `Распознанные типы логов` оформлена зеброй;
   - в `Разделы отчёта и источники` вместо `Не найдены` используется колонка `Найдены` с отметками ✅ / ❌.

8. Раздел `Устройства в архиве`:
   - HTML-отчёт показывает подтверждённые профили устройств в формате `Валидатор АСКП с ридером ОТИ/ТТ`;
   - профиль устройства строится только по фактам запуска валидатора (`device_boot_reports`) и evidence-строкам;
   - BM package-маркеры `mgt_nbs-oti/tt` выводятся отдельно как `Package-маркеры` и не подменяют физический ридер устройства;
   - `PaymentStart` поля `serial_number`, `rid`, `trxid` не создают самостоятельные устройства без подтверждённого запуска валидатора;
   - раздел добавлен в HTML manifest и AI context.

9. Привязка диагностик к устройствам:
   - `Скорость загрузки устройства` получает `device_context` по serial подтверждённого запуска;
   - `Выход НБС в работу после открытия смены` и `Долгое чтение и валидация карт` получают `device_context`;
   - если в загрузке ровно одно подтверждённое устройство, диагностические кейсы связываются с ним со статусом `single_confirmed_device`;
   - если подтверждённых устройств нет, diagnostics получают статус `unconfirmed` и текст `Устройство не подтверждено логами запуска`;
   - связи добавлены в HTML manifest и AI context.

10. Диагностика и группировка по `device_context`:
   - для архивов/диагностик без подтверждённых запусков добавлен пользовательский блок `Подтверждение устройств`;
   - блок показывает факт `Устройства не подтверждены логами запуска`, package-маркеры, количество сценариев НБС и кейсов чтения карт;
   - `Скорость загрузки устройства`, `Выход НБС в работу после открытия смены` и `Долгое чтение и валидация карт` получили таблицы группировки по `device_context.status` и `device_id`;
   - в manifest добавлены группы `device_boot_speed_device_groups`, `nbs_startup_device_groups`, `card_reading_speed_device_groups`;
   - для `kzp_logs_260715.zip` устройство не угадывается по package-маркерам: `device_profiles: 0`, diagnostics сгруппированы как `unconfirmed`.

## Контракт обработки файлов

Для каждого источника загрузки отчёт должен сохранять проверяемый lifecycle:

```text
найден в архиве -> извлечён -> распознан -> просканирован -> использован в разделах отчёта
```

Обязательные инварианты для archive input:

1. `log_file_count` считается по поддерживаемым log-файлам archive inventory с учётом рекурсивно распакованных вложенных архивов.
2. Если `log_file_count > extracted_file_count`, в `skipped_reasons` должна быть причина `не извлечены: ...`.
3. Если `extracted_file_count > analyzed_file_count`, в `skipped_reasons` должна быть причина `не сканировались: ...`.
4. Если файл не является log-файлом, он считается в `other_file_count` и получает причину `не анализировались: прочие файлы не относятся к log-файлам`.
5. `sum(log_type_counts.values())` должен совпадать с количеством log-файлов, тип которых выводится в блоке `Он содержит логи следующих типов`.
6. Для полностью поддерживаемого архива ожидается `log_file_count == extracted_file_count == analyzed_file_count`.
